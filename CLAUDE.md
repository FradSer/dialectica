# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

Dialectica is a reasoning-engine toolbox built on Google ADK. The shipped public API is now just two things, both proven by controlled evals: the **`Workflow` kernel** (`workflow.py` — `agent`/`parallel`/`pipeline`/`phase`/`log`/`budget`), whose `agent(tools=...)` primitive lets a stage do what a single forward pass cannot (act → observe → iterate; measured 8/8 vs a single call's 0/8 for a small model on tasks requiring interaction), and **execution-guided repair** (`repair.py`, verifier-in-the-loop: best-of-N reliability at ~1/3 the cost on verifiable tasks, now rebuilt on top of the kernel). Everything else this project built — a dedicated agentic-engine class, a heterogeneous ensemble, a dialectic (thesis → antithesis → synthesis) engine, and a legacy Tree-of-Thoughts + GAN baseline — was measured to either need nothing beyond `agent(tools=...)` or to tie/lose a prompt-matched single call as a pure-LLM scaffold, so all four were demoted to unshipped reference patterns in `examples/patterns/`. Hard-won finding: no pure-LLM scaffold beats a prompt-matched single call on *self-contained* tasks — engines win only by adding capability (tools) or ground-truth verification. Published on PyPI as `dialectica`. Python 3.11+.

## Commands

- Install deps: `uv sync`
- Test (mocked, fast, no API key): `uv run pytest`
- Test live E2E (calls real Gemini, needs `GOOGLE_API_KEY`): `uv run pytest -m e2e` — auto-skipped without the key
- Eval (engine vs single-call baseline, real API): `uv run python -m evals [--limit N --json out.json]`
- Format / lint: `uv run ruff format` / `uv run ruff check`
- Never hand-edit `pyproject.toml`; use `uv add` / `uv remove`.

## Testing workflow

BDD-driven TDD: new behavior starts with a Gherkin scenario in `tests/features/*.feature`, executable via pytest-bdd — step definitions live in `tests/test_*_feature.py` (bound with `scenarios(...)`). Then RED test → GREEN code → REFACTOR. When updating tests, update the matching `.feature` first. This rigor is reserved for the shipped kernel + repair; `examples/patterns/` gets a lighter regression net (`tests/test_example_patterns_smoke.py`, one mocked end-to-end run per pattern) since it's reference code, not shipped API.

Mock the LLM at the single seam `agent_runtime.run_agent()` — never patch ADK internals or per-stage agents (`tests/helpers.py` has the fakes). `asyncio_mode = auto`, so plain async tests need no decorator; pytest-bdd steps are sync — wrap coroutines with `asyncio.run()`. Tests load `dialectica/.env` via `tests/conftest.py`.

CI (`.github/workflows/test.yml`) runs `ruff format --check`, `ruff check`, and `pytest` on every push/PR; the release workflow runs the same gates.

## Architecture

**Two shipped things, honest hierarchy** (controlled evals in `evals/results/`, README "Evaluation"):

- `workflow.py` — `Workflow` + `agent`/`parallel`/`pipeline`/`phase`/`log`/`budget`: the execution kernel (a Python port of Claude Code's `Workflow` tool surface). `agent(prompt, *, schema=None, tools=None, instructions="", label=None, phase=None, model=None)` is one LLM call; `schema` forces structured JSON output via ADK `output_schema` (reuses `json_repair`'s fence/escape repair + re-ask). **`tools=[...]` is the one lever that turns a stage into a genuine capability-add** — the same wiring `examples/patterns/agentic_pattern.py` uses, measured **8/8 vs single-call 0/8** on the hidden-oracle benchmark (`evals/agentic_eval.py`). ADK forbids combining `tools` with `schema` on one call, so `agent()` raises `ValueError` if both are passed — split into a tool-using stage then a schema-only stage. `instructions` appends task-specific system-prompt framing; `model` resolves a `provider:model` override the same way every roster call site does. The concurrency cap (`DIALECTICA_WORKFLOW_CONCURRENCY` or `Workflow(concurrency=...)`, default min(16, cpu-2)) gates each `agent()` LLM call — `parallel`/`pipeline` branches idling in non-LLM work hold no slot. `budget` meters both `agent()` calls and API-reported token usage (`run_agent` returns an `AgentResponse`, a str subclass carrying `TokenUsage` summed from ADK `usage_metadata` — plain-str test fakes meter as zero); `Workflow(budget_total=..., budget_unit="tokens")` gates on output tokens (thinking included, the Workflow tool's unit) instead of call counts, and `spent_calls()`/`spent_tokens()`/`usage()` are always available. A workflow built entirely from schema-only judge/synthesize stages (no `tools`) is still a pure-LLM scaffold and still bound by the negative findings below. Resume/journaling NOT in v1 (documented gap).
- `repair.py` — `create_repair_engine`: verifier-in-the-loop (`generate → verifier → repair-on-failure`), rebuilt on top of `workflow.py`'s `agent(model=..., label=...)` inside a private `Workflow` script — no bespoke `LlmAgent` construction of its own. On verifiable tasks it *ties* matched-cost best-of-K on pass-rate but reaches it far cheaper (short-circuits: ~1/3 the calls). Verifier is an injected `Callable[[answer], (passed, feedback)]`; `solution_format` pins output shape; `models=[...]` rotates a roster round-robin on failure. Returns `{final_answer, passed, attempts, history}`. Called inside an outer `Workflow` script, `run()` joins that run's context (attempts charge the outer `budget_total`, the child-workflow rule); standalone it opens its own. Reproduce with `evals/repair_ablation.py`.

`agent_factory.py` builds ADK `LlmAgent`s from `ROLE_TEMPLATES` — trimmed to `{"Generator": ...}` only, since every shipped call site now uses `wf.agent(instructions=...)` instead of a Discriminator/Synthesizer role template. `agent_runtime.run_agent()` is the only place that actually calls the LLM; it retries transient failures with exponential backoff (`_call_agent_once` is the raw transport — retry tests patch that, everything else patches `run_agent`). `json_repair.py` holds the two shared fence/escape-repair helpers `workflow.py` depends on (all that survived from the former `gan_evaluator.py` once its GAN-specific classes were demoted). The library never calls `logging.basicConfig` — the consuming app owns logging setup.

**`examples/patterns/`** (a dev tool, not shipped in the wheel, same as `evals/`) holds runnable reference implementations of everything the evals did NOT justify shipping — each keeps its demoted engine's exact factory name/signature/return-shape, rebuilt on `workflow.py` primitives instead of bespoke agent construction, written in the kernel's own compositional idiom (plain functions/closures over `agent()`/`parallel()`) rather than a Protocol-based plugin system:

- `agentic_pattern.py` — `create_agentic_engine`/`AgenticEngine`: wraps `agent(task, tools=tools, instructions=AGENTIC_SYSTEM_PROMPT...)` in a `Workflow` script. Same 8/8 vs 0/8 result as the kernel primitive; kept as a copy-pasteable recipe with the tailored system prompt, not because the capability needs a class.
- `dialectic_pattern.py` — `create_dialectic_engine`/`DialecticEngine`: thesis → antithesis → synthesis spiral, scoring via `agent(schema=Verdict)` (`_scoring.py`'s shared `Verdict` model). Prompt-controlled eval showed it ties/loses a prompt-matched single call (**0-3-2** at every model size — it rearranges the model's own thinking, adding no information); positioned for content-steering (`criteria`) + auditable reasoning, not raw quality.
- `ensemble_pattern.py` — `create_ensemble_engine`/`EnsembleSearchEngine`: AB-MCTS-lite adaptive search (Thompson-sampling bandit) over a heterogeneous roster, mandatory float `scorer`. **CUT** by the honesty gate — a blind-pick roster (scorer replaced by a constant) matched the real scorer's robustness gain; the signal adds nothing over heterogeneity alone. FR6 roster-distinctness warnings were NOT ported (needed pre-built agents, which this pattern no longer has).
- `tot_gan_pattern.py` — `create_engine`/`create_coordinator`/`Coordinator`: beam search + GAN-style adversarial refinement; local `Node` dataclass + `active_beam` list instead of the original mutable `thought_tree`/Protocol system; sibling expand/evaluate concurrent via `wf.parallel`. **Measured dominated**; baseline only. The cross-call parse-failure circuit breaker and the `structured_output=False` enforced-JSON workaround were NOT ported (acceptable simplifications for study-only code).

Top-level `evals/` is the eval harness (dev tool, not shipped in the wheel): each benchmark problem (`problems.py`) is solved by the engine and by a single-call baseline (`baseline.py`), then a blind judge (`judge.py`) compares both answers twice with swapped positions — disagreement is a tie. `harness.py` counts LLM calls through the `run_agent` seam (its `EngineLike` structural type is what any engine — shipped or a demoted pattern — needs: just an async `run()`) and renders the report. Model overrides: `BASELINE_MODEL_CONFIG` / `JUDGE_MODEL_CONFIG`. `evals/*.py` import the demoted patterns from `examples.patterns.*` — e.g. `evals/quality_ablation.py --engine tot` still runs `examples/patterns/tot_gan_pattern.py`, preserving the historical numbers below.

Eval findings (2026-06, three 3-run × 5-problem matrices, see README "Results"): with the original "Innovation" discriminator criterion (V1) the engine won technical problems 7-1-1 but lost organizational ones 0-4-2 ("over-complex"), reproducing across Gemini and Qwen. Replacing "Innovation" with "Feasibility under stated constraints" (V2 + second seed V3) pooled to 20-8-2 (technical 15-2-1, organizational 5-6-1) vs V1's 7-5-3. Key insight: because the GAN loop refines thoughts against the critique, discriminator criteria steer answer content, not just selection. Engine cost ≈ 20-30× baseline calls; concurrency cut wall-clock 2-3× at unchanged call counts. Reports land in `evals/results/` (gitignored).

Prompt-controlled finding (2026-06-15, qwen3.6-35b-a3b via CLIPROXYAPI, blind position-swap) — **why repair is the core**: strengthening the dialectic (synthesis "dominate a single expert pass" + thesis baseline-strength) beat a *naively*-prompted single call (4-1-0) and plain ToT (pooled 8-2-0), but against a **prompt-matched** strong baseline it went **0-3-2 (loses) at equal length** — the gain was the prompt, not the structure. Lesson: pure-LLM scaffolds (ToT, GAN, dialectic) only rearrange the model's own thinking on one context, so they tie a single call; a scaffold beats one pass only by adding information a single pass lacks — tool-grounding (`repair.py`), scale, or independence. On verifiable tasks the band where repair beats a single call is *fails-but-fixable*, and it shrinks as the model strengthens (this model solved HumanEval 17-18/18 and curated LCB-hard at pass@1, leaving little gap to demonstrate).

SWE-suite findings (2026-06, ground truth): against a 1-attempt baseline the engine looked +1 (11/12 vs 10/12 on gpt-oss:20b; 10/12 vs 9/12 on gemma4:e4b), but rescue mode with a pass@2 screen dissolved it — the apparent lift was resampling luck, and the single real gap (max-fill on e4b) was NOT rescued (full mode even regressed min-path on e4b). After extending to 18 problems with the literature's hardest HumanEval items, cloud gemma-4 (31b-it and 26b-a4b-it via AI Studio) solved 18/18 at pass@2 — HumanEval-class is saturated for this family; real code-value evidence needs LiveCodeBench/competition difficulty. Methodology rules: compare scaffolds against pass@k at matched cost; the advice suite still needs a best-of-2-plus-judge control. Backend quirk: enforced JSON mode breaks gemma-4-26b-a4b-it (empty/truncated verdicts) — the ToT+GAN pattern's `structured_output=False` toggle for this is documented but not actually wired to anything after the demotion (see pattern docstring).

2026-06-30 consolidation (this file's own honesty gate): Agentic (75 LOC) and Repair (236 LOC) were the only two ✅-verified wins; `AgenticEngine`'s unique logic was ~15-19 lines (a system prompt + a name), and `create_agent()`/`workflow.agent()` already supported `tools=` generically — so Agentic was folded entirely into the kernel primitive rather than kept as a separate class. Ensemble (already ⚠️ CUT) and ToT+GAN (already ❌ dominated) were demoted alongside it to `examples/patterns/`, since neither was pulling weight as bespoke shipped engines. `workflow.agent()` gained `instructions=` (to preserve Agentic's system-prompt capability) and a fix for a latent bug where `model=` was never resolved through `_parse_model_config` (found during the port, never previously exercised since no test or eval called `wf.agent(model=...)`).

## Gotchas

- The library does **not** load `.env` — consuming apps own that. Only the test suite loads it (`dialectica/.env`).
- Models: use `gemini-3.5-flash` (default) or `gemini-3.1-pro-preview` only — there is no stable `gemini-3.1-pro` (404 on generateContent). Configure via `provider:model_name` env vars (`DEFAULT_MODEL_CONFIG`, plus optional `GENERATOR_MODEL_CONFIG` — the only role-specific override the shipped kernel reads, since `ROLE_TEMPLATES` only has `Generator` now). Parsed in `llm_config.py`.
- Env vars: `GOOGLE_API_KEY` for AI Studio, or the Vertex trio (`GOOGLE_GENAI_USE_VERTEXAI=true`, `GOOGLE_CLOUD_PROJECT`, `GOOGLE_CLOUD_LOCATION`). See `.env.example` (repo root; copy to `dialectica/.env` for the test suite).
- `examples/patterns/*.py` are dev-tool reference code, not shipped in the wheel (`pyproject.toml` only packages `["dialectica"]`) — import them the way `evals/*.py` does (`from examples.patterns.agentic_pattern import create_agentic_engine`, etc.), not from the `dialectica` package.
